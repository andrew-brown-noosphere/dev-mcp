# Why Your PLG Strategy is Struggling in the AI Era

*November 28, 2025*

Product-Led Growth was designed for humans exploring your product. AI agents don't explore—they evaluate. If your PLG motion assumes human-driven discovery, you're optimizing for a funnel that's increasingly being bypassed.

## The PLG Playbook

Product-Led Growth transformed how developer tools go to market. The playbook is well-established:

- **Free tier** drives awareness and lowers friction
- **Self-serve onboarding** reduces customer acquisition cost
- **Product usage** drives expansion revenue
- **Viral loops** create organic growth

This works because developers discover your product, try it themselves, and adopt it bottom-up before procurement gets involved. The motion depends on developers finding you, signing up, and experiencing value through hands-on exploration.

## The Developer Journey is Changing

Caroline Lewko's Developer Journey Map describes five stages: **Discover, Evaluate, Learn, Build, Scale**. Traditional PLG optimizes for each stage—landing pages for discovery, free tiers for evaluation, documentation for learning, SDKs for building, enterprise features for scaling.

But the developer journey has evolved through four distinct eras:

**Era 1: Documentation (1995-2010)**
Discovery happened through books, conferences, and vendor documentation. Time to adoption measured in weeks to months.

**Era 2: Search (2005-2018)**
Google, Stack Overflow, and technical blogs became the discovery layer. Time to adoption dropped to days or weeks.

**Era 3: Social (2015-2024)**
GitHub stars, Twitter threads, and Discord communities drove adoption. Time to adoption fell to hours or days.

**Era 4: Agentic (2024-Present)**
AI assistants—Claude, ChatGPT, Cursor—increasingly mediate discovery. Time to adoption can now be measured in minutes.

In Era 4, the Lewko stages collapse. Discover, Evaluate, and Learn happen simultaneously within a single agent conversation. A developer asks "what database should I use for time-series IoT data?" and the agent recommends, explains, and helps implement—all before the developer ever visits your landing page.

## Why Traditional PLG Struggles

### Agents Don't Browse

The traditional PLG funnel assumes:
1. Developer finds your landing page
2. Signs up for free tier
3. Follows getting started guide
4. Builds something, experiences value
5. Upgrades or refers others

The agentic reality looks different:
1. Developer asks AI for recommendations
2. Agent evaluates options (if MCP-accessible)
3. Agent recommends and helps implement
4. Developer may never see your landing page

### The Invisible Product Problem

If agents can't access your product:

- **No discovery**: You're not in the consideration set when agents make recommendations
- **No evaluation**: The agent can't test your product, so it can't recommend with confidence
- **No differentiation**: Without hands-on evaluation, agents rely on training data and documentation—often outdated

You've optimized for a funnel that's being bypassed by developers who now ask AI first.

## The Agentic Technology Adoption Lifecycle

Agent-mediated adoption follows a new pattern:

1. **Agent Awareness**: Is your product in training data? Do you have MCP servers? Is your documentation agent-readable?

2. **Agent Evaluation**: Can the agent actually test your product? Connect to APIs? Run benchmarks? This is where most products fail—they're discoverable but not evaluable.

3. **Agent Recommendation**: Given the developer's specific use case, does your product win on merit? Agents compare options based on actual capabilities, not marketing.

4. **Human Validation**: The developer verifies the agent's recommendation. This is still critical—trust is established, not delegated.

5. **Agentic Scaling**: Production deployment with ongoing agent assistance for optimization, migration, and troubleshooting.

## Adapting PLG for the Agentic Era

The fix isn't abandoning PLG—it's layering agentic accessibility onto your existing motion.

### Make Your Product Agent-Evaluable

**MCP Servers** let agents actually use your product. Not just query documentation—execute operations, run tests, explore capabilities. When an agent can connect to your database, it can recommend with confidence.

**Demo Environments** enable meaningful evaluation. Sandboxed instances where agents can deploy sample apps, run benchmarks, and generate real comparisons.

**llms.txt and context.txt** ensure proper discovery. These files help AI understand your product's positioning, use cases, and differentiators—structured for agent consumption.

### The New Metrics

| Traditional PLG | Agentic PLG |
|-----------------|-------------|
| Website visits | Agent queries |
| Signups | MCP connections |
| Activation | Integration success |
| Retention | Continued agent use |
| Referral | Agent recommendations |

### Enterprise PLG Without Free Tiers

For enterprise products that can't offer self-serve free tiers, agentic accessibility provides an alternative:

- **MCP servers with demo environments** let agents evaluate without full product access
- **Agent-accessible benchmarking** provides evidence-based comparison
- **Comparison tools that work programmatically** enable side-by-side evaluation

Your product can "smell like PLG" to agents even without a free tier—by being evaluable rather than just discoverable.

## The Path Forward

1. **Audit your current PLG funnel** — Where do developers enter? What's the onboarding path?

2. **Map the agentic journey** — Where might developers ask AI instead of browsing? What recommendations are agents currently making?

3. **Add agentic layers** — MCP servers, llms.txt, demo environments. Make evaluation possible, not just discovery.

4. **Measure both funnels** — Track traditional metrics alongside agent queries, MCP connections, and agent-attributed conversions.

5. **Optimize for agentic discovery** — The new DevRel includes maintaining MCP servers and agent-readable context.
