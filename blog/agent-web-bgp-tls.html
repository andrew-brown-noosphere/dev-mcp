Why the Agent Web Will Look More Like BGP + TLS Than Sitemaps + Robots.txt

In my previous post, I argued that search indexes have become the de facto DNS layer for AI agents. Agents don't navigate—they're routed through whatever the search index surfaces.

That post identified the problem. This one explores what the solution might look like.

The short version: The infrastructure agents need will resemble BGP and TLS more than sitemaps and robots.txt. It's not about better metadata. It's about routing, identity, and trust.

The Response to the DNS Problem

After publishing the DNS piece, I saw a predictable response: "We're solving this with llms.txt / schema markup / MCP manifests / OpenAPI specs."

These are useful artifacts. But they share a fundamental limitation:

They are descriptive. They describe content.

They don't solve routing.
They don't solve authority.
They don't solve trust.

To understand what's actually needed, we have to look at two foundational internet protocols that most people never think about.

BGP: The Internet's Invisible Routing Spine

When you visit a website, your packets don't magically know where to go. They traverse multiple networks—your ISP, regional carriers, cloud providers, CDNs—owned by entirely different organizations.

How do these networks coordinate?

BGP (Border Gateway Protocol).

BGP allows large networks (Autonomous Systems) to announce their reachability:

"I can reach this IP range. Route traffic through me."

Other networks decide whether to trust that announcement and propagate it.

BGP answers a foundational question: Who is authorized to route traffic for a given destination?

It's not about content. It's about authoritative reachability.

Without BGP, the internet fragments into islands.

The Agent Parallel

In my previous post, I noted that agents currently depend on search indexes for discovery. But search isn't routing—it's discovery by inference.

When agents become primary actors—executing transactions, not just answering questions—inference won't cut it. They'll need to know:

  • Is this the canonical endpoint for this capability?
  • Is this source authoritative?
  • Who vouches for this route?

That's a BGP-shaped problem.

TLS: Cryptographic Identity

Once packets reach a server, a different problem emerges: How do you know you're talking to who you think you're talking to?

TLS does three things:

  1. The server proves its identity via certificate
  2. The client verifies against a trusted root authority
  3. Both sides establish an encrypted channel

TLS answers: Is this endpoint really who it claims to be?

Without TLS, you can reach an endpoint but you can't trust it.

The Agent Parallel

Agents can't visually verify anything. They can't notice that paypa1.com isn't paypal.com. They can't recognize a phishing page by its typography.

When an agent executes consequential actions, it needs cryptographic certainty:

  • Who is on the other end?
  • Was this response tampered with?
  • Is this entity authorized to act?

Today's agent architectures largely punt on this. They trust that search results lead to legitimate endpoints.

As I wrote before: agents trust whatever search returns. That's not infrastructure. That's a prayer.

Why Static Declarations Aren't the Answer

The current approach to making the web "agent-friendly" focuses on metadata:

  • sitemaps — pages that exist
  • schema.org — structured content data
  • robots.txt / llms.txt — access guidance
  • OpenAPI — API documentation
  • MCP manifests — tool capabilities

These are all static declarations. They say:

"Here is what I expose."

They don't answer:

  • Who authorizes this endpoint?
  • Is this the canonical source, or a mirror, or a fake?
  • Can an agent rely on this for consequential decisions?
  • What policies govern automated action?

They're fragments. Not a handshake.

In the previous post, I called this the "Gate 1 vs Gate 2" problem: these files solve understanding (Gate 2) before discoverability (Gate 1). But the problem is deeper than that.

Even if agents could discover these files, the files don't establish trust. A sitemap tells you pages exist. It doesn't tell you whether the entity is legitimate or whether an agent is authorized to act.

llms.txt is closer to a README than to a certificate.

The Missing Control Plane

If agents become primary web actors, we'll need a new control plane that:

  1. Resolves authoritative endpoints — not through search, but direct resolution
  2. Verifies identity cryptographically — endpoints prove who they are
  3. Exposes machine-readable capabilities — what can be done, under what constraints
  4. Enforces executable policy — governance rules that execute, not just publish
  5. Negotiates action rights — both sides agree before execution

This is structurally closer to BGP + TLS than to SEO.

It's infrastructure. Not metadata.

The Capability Shift

The current web is page-centric. Everything revolves around documents humans read.

Agents operate in a capability-centric world. They don't want to read your About page. They want to know:

  • What can I do here?
  • What do I need to provide?
  • What will I get back?
  • Who vouches for this?

Instead of:

navigate → interpret → decide

We're moving to:

resolve → verify → negotiate → execute

This changes where power sits:

  • Not in content aggregation
  • Not in ranking algorithms
  • But in authoritative resolution and trust infrastructure

What This Might Look Like

Imagine:

  1. Direct capability resolution — An agent needing to book a flight doesn't search "airline booking." It resolves flights.delta.com through a routing protocol, receiving a signed capability manifest.
  2. Cryptographic credentials — The booking API presents a certificate chain proving authorization, issued by a verifiable trust authority.
  3. Negotiated capabilities — Before executing, agent and endpoint exchange a machine-readable contract: required data, permitted actions, applicable policies.
  4. End-to-end provenance — Every piece of context carries attestations: who created it, who reviewed it, what evidence supports it.
  5. Executable governance — Rate limits, authorization scopes, audit requirements encoded in protocol, not ToS.

The components exist: W3C PROV-O for provenance, C2PA for content authenticity, in-toto for attestation, DIDs for decentralized identity.

What's missing is the integration layer.

The Browser Era vs. The Agent Era

The browser era optimized for human navigation: render pages, make links clickable, trust the human to judge.

The agent era will optimize for machine coordination: resolve capabilities authoritatively, verify identity cryptographically, negotiate programmatically, enforce policy automatically.

The primitives are different. The trust models are different.

Building the agent web on sitemaps and schema markup is like building e-commerce on Gopher. The foundation isn't wrong—it's insufficient.

Conclusion

In the previous post, I argued that search indexes have become an unintentional DNS layer for agents—and that this creates fragility.

The solution isn't better metadata layered on top of the human web.

It's the infrastructure layers the human web could avoid—because humans compensate with judgment and caution.

Agents can't compensate. They need:

  • Routing — authoritative paths to capabilities
  • Identity — cryptographic proof
  • Trust — verifiable provenance
  • Governance — executable policy

That's not llms.txt. That's BGP + TLS for the agent era.

The organizations that build this infrastructure—or position themselves as trust anchors within it—will shape how agents interact with the world.

Everyone else will be routed around.
